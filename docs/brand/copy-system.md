## 0. 能力边界

文案发布前，必须先区分“已经可以操作的产品能力”和“仍需开发或验证的规划能力”。首页可以展示完整产品方向，但不得把规划能力伪装成已经上线。

### 0.1 已实现，可直接使用“支持”“可以”“已记录”

- 多组织工作空间、成员账号与角色权限。
- 品牌故事、表达风格和农产品档案的创建、编辑与组织隔离。
- TXT、Markdown、CSV 文本资料导入；资料保留来源文件名、类型、引用标签与文本摘要。
- 知识草稿、提交审核、通过或驳回，以及不可覆盖的线性修订记录。
- 仅从已审核知识中选择生成上下文。
- 当前基于确定性词法规则的有限上下文选择；记录来源、摘录摘要、字符数和截断状态。
- 30 秒和 60 秒短视频脚本、直播开场、单品讲解、直播互动、评论回复、社交文案、标题及封面文案生成。
- 内容任务维护、生成记录、模型与 Prompt 版本、引用来源和耗时记录。
- 人工修改形成新版本，显式提交审核，通过或驳回后保留完整历史。
- 已审核版本的发布登记，以及浏览、点赞、评论、分享、收藏、涨粉、订单、收入等原始数据快照。
- 以人工证据为基础的视频诊断记录。
- 从诊断建立改进 Brief，并由人明确创建后继内容草稿。
- 组织审计记录、Windows/SQLite 本地运行和 Docker/PostgreSQL 运行方式。

### 0.2 规划中，只能使用“计划”“将支持”“正在建设”

- PDF、DOCX、PPTX 的自动解析与结构化提取。
- 视频文件自动识别、画面理解、语音转写和自动诊断。
- 对接真实商业大模型或由用户自行配置的外部模型 API。
- 与抖音、小红书、视频号等平台自动同步发布数据。
- 更丰富的内容日历、多人协作提醒和跨渠道运营视图。
- 独立 Windows 可执行安装包；当前验证方式仍需要 Python 3.12。

### 0.3 不得承诺

- 爆款预测、销量保证、收益提升百分比或跨平台“统一高分”。
- 自动证明资料真实；SHA-256 只能证明提交文本是否发生变化。
- AI 自动代替审核人承担事实、合规或发布责任。
- “语义 RAG”“向量知识库”“智能检索”等与当前实现不符的技术描述。当前应称为“经审核知识的有限上下文选择”或“确定性词法选择策略”。
- VR、医疗功能、数字人直播、联邦学习、区块链或 NFT。

## 1. 一句话定位

### 1.1 推荐主定位

**禾语 AI，把经过确认的农产品知识，变成有来源、可审核、能持续改进的内容。**

这句话同时回答三个问题：

1. 输入是什么：经过确认的农产品知识。
2. 输出是什么：可用于传播的内容。
3. 与普通生成工具的区别是什么：有来源、可审核、能改进。

### 1.2 三种语气版本

#### A. 品牌型：温度与记忆

**让每一份土地里的认真，都被准确地讲出来。**

适合：首页首屏、品牌影片、比赛开场。  
注意：它是品牌表达，不应单独承担产品功能说明；下方必须补充功能副标题。

#### B. 产品型：明确与可信

**从可信资料到可发布内容，一套工作台完成建档、生成、审核与复盘。**

适合：产品介绍、工作台登录页、功能总览。

#### C. 商业型：价值与持续运营

**沉淀品牌事实，稳定内容生产，让每一次发布都成为下一次改进的依据。**

适合：路演、合作方介绍、商业计划书。

### 1.3 极短版本

- **有据可讲，有序共创。**
- **让好产品，好好被看见。**
- **从一方水土，到一方表达。**
- **内容有来源，表达有分寸。**

极短版本只用于视觉留白较大的品牌位置，不代替完整定位。

## 3. 首页分区文案

以下结构覆盖当前已经实现的完整工作流，也为规划能力留下扩展位置。

### 3.1 首屏后的价值证明

**眉题**  
为什么是禾语 AI

**标题**  
生成只是一步，可信才是一整套过程。

**说明**  
普通工具交付一段文字，禾语 AI 保留这段文字从哪里来、经过谁确认、如何修改，以及发布后为什么继续调整。

**四项短标签**

- 资料经人工确认
- 生成来源可追溯
- 内容版本不覆盖
- 运营记录可回看

**CTA**

### 3.2 用户痛点分区

**眉题**  
真实工作中的断点

**标题**  
资料散在各处，内容每次重来。

**说明**  
农产品信息常散落在表格、文档和聊天记录里；创作者不熟悉产品，审核者也难以判断一句话从何而来。禾语 AI 先把事实变成组织资产，再让 AI 参与表达。

**痛点卡片**

1. **资料难复用**  
   产品参数、品牌故事和禁用表达缺少统一入口。
2. **生成难核实**  
   AI 写得快，却很难确认依据和边界。
3. **协作难交接**  
   修改覆盖旧稿，审核意见与最终版本脱节。
4. **运营难沉淀**  
   发布数据留在平台，下一轮创作仍从感觉出发。

**CTA**

### 3.4 工作流程分区

**眉题**  
一条清楚的内容生产链

**标题**  
事实有人确认，表达由人定稿。

**步骤**

1. **建立档案**  
   录入品牌故事、产品参数、卖点和表达边界。
2. **审核知识**  
   确认资料来源，提交审核，保留每次修订。
3. **发起创作**  
   选择平台、受众、目标和内容类型，生成结构化初稿。
4. **修改定稿**  
   人工调整内容，形成新版本并进入审核流程。
5. **登记发布**  
   只登记通过审核的版本，持续保存真实运营数据。
6. **复盘改进**  
   用观察和证据建立改进 Brief，再开启下一版创作。

**CTA**  
亲自走一遍流程

### 3.5 可信机制分区

**眉题**  
Trust by design

**标题**  
可追溯，不等于堆技术名词。

**说明**  
禾语 AI 不用“智能”掩盖不确定性。系统清楚区分原始资料、人工审核、AI 生成、人工修改和发布观察，让团队知道每一步是谁做的、依据是什么。

**机制标签**

- 组织数据相互隔离
- 权限由服务端执行
- 未审核知识不进入生成
- 生成上下文有数量与长度边界
- 修改形成新版本
- 审核结果不可静默覆盖
- 发布指标保留时间快照
- 不提供未经验证的爆款预测

## 4. 工作台文案体系

工作台不是第二个营销首页。用户登录后，应优先看到当前状态、待办事项和下一步行动，而不是大幅品牌口号。

### 4.1 工作台首页

**页面标题**  
经营概览

**欢迎标题候选**

- 今天，先完成哪一步？
- 工作空间概览
- 从待办开始

推荐使用“今天，先完成哪一步？”，但字号应与功能页标题同一层级，不作为巨型 Hero。

**说明**  
查看内容资产、待审核事项和最近进展，继续完成下一项工作。

**动态下一步提示**

- 尚无品牌：**先建立品牌档案，统一团队的基础表达。**
- 尚无产品：**为品牌添加第一项农产品资料。**
- 尚无知识：**录入一条可核实的产品事实。**
- 存在知识草稿：**提交资料审核，审核前不会进入内容生成。**
- 存在待审核知识：**有资料等待确认，审核后才能用于生成。**
- 尚无内容任务：**选择一个产品，建立第一项内容任务。**
- 存在待审核内容：**有内容版本等待审核。**

### 4.6 内容审核

**页面标题**  
内容审核

**页面说明**  
确认事实、表达和发布边界；审核只改变当前版本状态，不覆盖任何历史内容。

**空状态**

标题：**当前没有待审核内容**  
说明：创作者提交新版本后，会出现在这里。  
CTA：**查看全部内容任务**

**审核检查提示**

1. 产品事实是否与引用资料一致？
2. 是否出现禁止表述或无法证明的承诺？
3. 语气是否符合品牌表达？
4. 内容结构是否适合目标平台和受众？

- 提交审核
- 通过版本
- 驳回版本
- 填写审核意见

- 查看引用来源

- 提交：**提交后，这个版本将进入待审核状态，不能再直接修改。确认提交吗？**
- 通过：**通过后，该版本可以登记发布。确认采用这个版本吗？**

**三个子区**

1. **发布记录**  
   记录平台、发布时间、外部链接或内容 ID。
2. **数据快照**  
   保存某一时间点的原始表现，不覆盖过去记录。
3. **诊断与改进**  
   记录观察、证据、风险与机会，并建立后续 Brief。

**空状态**

**边界提示**

- **这里展示原始记录，不计算未经验证的“爆款分数”。**
- **诊断是团队基于证据作出的判断，不是自动效果预测。**

## 5. 比赛路演叙事

### 5.1 30 秒版本

很多优质农产品不缺故事，缺的是把真实信息稳定讲清楚的能力。资料散在文档和聊天记录里，创作者不了解产品，普通 AI 生成又难以核实。禾语 AI 把品牌故事、产品事实和来源资料沉淀为经过审核的知识，再生成有引用、可修改、可审核的短视频与直播内容。发布后，团队还能记录真实数据和诊断证据，让下一次创作不再从零开始。我们不是替农业编故事，而是帮助真实产品被准确、持续地表达。

### 5.2 90 秒版本

一款农产品从田间走向市场，往往不是因为产品不够好，而是因为信息没有被整理，也没有被稳定地表达。产地、规格、种植方式、品牌故事散落在表格、文件和聊天记录里；换一个创作者，内容就要重新了解一遍。使用通用 AI 虽然可以更快写出文案，却很难回答：这句话的依据是什么，谁确认过，修改之后还能不能找回原来的版本？

禾语 AI 从这个真实断点出发。团队先建立品牌和农产品档案，再录入有来源的资料。资料需要经过人工审核，只有确认后的知识才能进入内容生成。创作者可以围绕平台、受众和传播目标，生成短视频脚本、直播话术、评论回复、社交文案和标题封面，并查看本次生成采用了哪些来源。

AI 生成不是终点。人工修改会形成新版本，不覆盖历史；内容需要显式提交并通过审核后，才能登记发布。发布后，团队可以持续保存真实运营数据，记录有证据的视频诊断，再把问题和机会转成下一轮改进 Brief。

因此，禾语 AI 不只是一个脚本生成器，而是一套从事实沉淀、内容生产到运营改进的农业内容工作台。我们不承诺爆款，也不替代人的判断；我们希望让每一份土地里的认真，都有依据地被讲清楚，并成为组织能够长期积累的内容资产。

### 5.3 路演画面对应

| 时间 | 画面 | 讲述重点 |
| --- | --- | --- |
| 0–12 秒 | 农产品、产地与散落资料 | 产品好，但信息难以稳定传达 |
| 12–25 秒 | 品牌与产品档案 | 把分散事实沉淀为统一资产 |
| 25–40 秒 | 知识提交与审核 | 先确认，后生成 |
| 40–57 秒 | 多类型内容生成及来源 | 内容更快，同时知道依据 |
| 57–70 秒 | 版本比较与人工审核 | 人保留最终决定权 |
| 70–82 秒 | 发布快照、诊断、Brief | 用真实证据推动下一次创作 |
| 82–90 秒 | 品牌收束画面 | 不是编故事，而是讲清真实产品 |

## 6. 禁用措辞

### 6.1 空泛的 AI 味表达

以下词语并非绝对不能出现，但如果不能说明具体动作，应删除或改写：

- AI 赋能农业
- 数智化新范式
- 颠覆传统
- 重塑行业生态
- 全链路智慧升级
- 一站式智能解决方案
- 深度赋能
- 多维协同
- 智慧引擎
- 内容生产新纪元
- 打造闭环
- 精准触达
- 千人千面
- 降本增效
- 无限可能
- 让科技更有温度
- 以创新驱动高质量发展

**改写原则**

- “AI 赋能内容生产” → “从已审核资料生成短视频和直播初稿”
- “构建全链路闭环” → “连接建档、审核、创作、发布和复盘”
- “实现精准触达” → “围绕指定平台、受众和传播目标建立内容 Brief”
- “全面降本增效” → “减少重复查找资料和从零起稿的工作”

### 6.2 夸大或无法证明的表达

禁止：

- 一键生成爆款
- 保证销量增长
- 转化率提升 X%
- 全自动运营
- 零错误
- 绝对真实
- 杜绝幻觉
- 行业第一、国内领先、全球首创
- 最懂农业的 AI
- 自动发现最佳营销策略
- 精准预测传播效果
- 一次配置，永久有效
- 无需人工审核

推荐替代：

- “生成内容初稿”
- “减少重复起稿”
- “仅从已审核知识中选择上下文”
- “保留引用与生成记录，便于人工核对”
- “以真实运营记录支持后续判断”

### 6.3 技术误导

在当前实现下不得使用：

- 语义 RAG
- 向量知识库
- 多模态视频大模型诊断
- 自动语义检索
- 智能 Agent 自主运营
- 区块链溯源
- 数据真实性认证

当前准确表述：

- **经审核知识的有限上下文选择**
- **确定性词法选择策略**
- **来源与摘录指纹记录**
- **证据驱动的人工视频诊断**
- **Provider-neutral AI gateway**
- **本地确定性演示 Provider**

### 6.4 生硬对仗

不要为了整齐而牺牲自然表达，例如：

- 让农业更智慧，让智慧更农业。
- 以数据赋能生产，以智能重构传播。
- 上承乡村振兴，下启数字未来。
- 一边是田野，一边是世界；一边是产品，一边是梦想。

对仗应满足：

1. 两句各自有明确含义。
2. 共同推进同一个逻辑，而非同义反复。
3. 读起来像人会说的话。
4. 不借宏大政策语言掩盖具体产品价值。

可采用：

- 让好产品说得清，让真故事传得远。
- 有所依据，才敢表达；留有记录，才能改进。
- AI 负责提速，人负责定夺。
- 内容可以更快，事实不能含糊。

## 7. 已上线与规划中功能的标签规则

### 7.1 状态标签

只使用以下五类状态，避免“即将上线”“敬请期待”等无法核验的模糊承诺。

| 标签 | 使用条件 | 示例 |
| --- | --- | --- |
| **已上线** | 当前界面和 API 已可完整操作 | 品牌与产品维护 |
| **已上线 · 本地演示** | 流程已打通，但默认使用确定性演示 Provider | AI 内容生成 |
| **已上线 · 手工录入** | 数据结构与流程已上线，数据需用户输入 | 运营数据、视频诊断 |
| **Beta** | 已可使用，但体验或接口仍可能明显调整 | 仅在发布说明明确指定时使用 |
| **规划中** | 当前无法操作，只有产品方向或设计 | PDF 解析、自动视频理解 |

### 7.2 句式规则

#### 已上线

使用现在时和明确动作：

- “支持创建并维护品牌档案。”
- “仅通过审核的资料可以进入生成。”
- “每次人工修改保存为新版本。”

不要写：

- “致力于实现……”
- “将帮助用户……”
- “有望支持……”

#### 规划中

使用未来边界，说明当前替代方式：

- “计划支持 PDF 结构化提取；当前可导入 TXT、Markdown 和 CSV。”
- “计划增加自动视频理解；当前诊断由团队基于证据手工记录。”
- “真实模型 API 将按部署方配置接入；当前默认使用本地演示 Provider。”

不要写：

- “即将上线”，除非已有明确版本与发布日期。
- “已具备底层能力”，如果用户当前不能使用。
- “无缝接入”，如果尚未完成集成验证。

### 7.3 首页展示规则

1. 已上线能力可放在主流程中，用实心卡片或主色标签。
2. 规划能力应单独放入“正在建设”或“产品路线”分区，不与已上线功能混排。
3. 如视觉必须混排，每张规划卡必须在标题附近显示“规划中”，不能只在页面底部统一声明。
4. CTA 必须与状态对应：
   - 已上线：进入、体验、建立、查看。
   - 规划中：了解规划、查看路线、订阅进展。
5. 不使用不可点击的“立即体验”指向规划功能。

### 7.4 路演表达规则

- 可以展示规划蓝图，但讲述时必须使用：“下一阶段我们计划……”
- Demo 画面只能演示实际存在的交互。
- 若使用概念动效表示未来能力，画面应标注“Concept / 规划中”。
- 所有量化结果必须有真实试点数据、样本定义和统计口径；在此之前只描述流程价值，不描述效果百分比。

## 8. 文字视觉层级建议

本节不是视觉规范，只约束文案在界面上的信息密度。

### 8.1 首页

- Hero 主标题：最多两行，每行建议 8～14 个汉字。
- Hero 副标题：40～72 个汉字，最多两句。
- 卡片标题：4～10 个汉字。
- 卡片说明：28～55 个汉字。
- 同一屏只保留一个主 CTA。
- 英文眉题只作辅助，不得比中文信息更醒目。

### 8.2 工作台

- 页面标题：4～8 个汉字，不使用营销口号。
- 页面说明：一句话说明当前模块能完成什么。
- 面板标题：4～12 个汉字。
- 表单标签：优先使用名词，不写成长句。
- 辅助说明不得小到只能作为装饰；如果信息重要，就应该具备正常可读性。
- 空状态由“当前状态 + 为什么 + 下一步”三部分组成。
- 按钮使用动作词开头，如“保存品牌”“提交审核”“建立新版本”。
