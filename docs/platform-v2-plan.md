# 禾语 AI 商业级 MVP 建设规划

> 更新日期：2026-07-15
> 状态说明：`已完成` 表示当前代码与自动化验收已有证据；`下一步` 表示尚未完成，不应作为现有能力宣传。

## 1. 产品主线

禾语 AI 面向缺少新媒体运营能力的农户、合作社和乡村运营团队，把普通农产品资料转化为可以拍摄、发布和持续执行的内容经营方案。

核心流程：

```text
选择身份与经营目标
→ 录入农产品资料
→ 选择发布平台
→ AI 整理产品信息
→ 生成平台差异化策略
→ 生成三套短视频、手机分镜与背景乐方向
→ 生成直播话术
→ 形成 7 天运营计划
→ 根据真实反馈进入下一轮优化
```

知识库、版本、审核、权限和运营记录是平台的长期资产层与系统护栏，不取代“帮助农户生产和经营内容”这一主定位。

## 2. 已完成的商业级 MVP 基础

- `已完成` 每条视频的标题、开头、正文、封面文案、背景乐方向、手机分镜与拍摄提示；
- `已完成` 直播讲解结构；
- `已完成` 7 天发布与运营计划；
- `已完成` 当季番茄、高山云雾茶、岭南当季水果礼盒三个一键案例；

### 2.2 模型运行层

- `已完成` 零成本、确定性的 Mock Provider，未配置密钥时仍能完成演示；

## 3. 当前架构

```text
品牌首页
├── 农户简单模式 /create
│   ├── 身份与目标
│   ├── 产品与受众
│   ├── 平台与手动热点
│   └── 策略、视频、直播与 7 天计划
└── 团队专业模式 /workspace
    ├── 品牌与产品资产
    ├── 产品资料与知识库
    ├── 内容项目与版本
    ├── 团队审核
    └── 运营复盘

业务 API
├── Marketing Workflow
├── Content Generation
├── Knowledge Retrieval
├── Quality Guardrails
└── Operations Planning

模型适配层
├── OpenAI-compatible Provider
├── 有界 TTL Cache
└── Deterministic Mock / Mock Fallback

数据层
├── SQLite（默认本地运行）
└── PostgreSQL（部署可选）
```

## 4. 后续实施次序

### P1：真实模型质量闭环

1. 选定一家国内可用的 OpenAI-compatible 模型服务进行真实接入验收；
2. 建立至少 3 类农产品、12 条任务的脱敏评测集；
3. 检查平台适配度、可拍摄性、自然度、事实一致性、三语质量、延迟和成本；
4. 保存失败案例并加入回归测试；
5. 增加管理员模型连接测试，但密钥只保存在服务端。

### P2：内容生产体验

1. 支持用户自由选择视频、直播、7 天计划等内容模块；
2. 支持单个模块局部重新生成；
3. 将产品资料和品牌语气更顺畅地复用于多次任务；
4. 增加适合不同平台的复制、导出和交付格式；
5. 在真实用户测试中验证农户能否在 10 分钟内完成一次任务。

### P3：热点与运营闭环

1. 接入合规热点来源，记录来源、抓取时间和有效期；
2. 对热点去重、聚类，并匹配产品、受众、目标与平台；
3. 建立内容日历和任务完成状态；
4. 支持 CSV/Excel 导入播放、互动、评论和销售线索；
5. 根据真实反馈生成复盘与下一轮内容建议。

当前热点仅支持用户手动输入，平台不会宣称已经抓取实时热点，也不会虚构热度。

### P4：知识库与商业运行

1. 文档切分与 Embedding；
2. 向量存储、关键词与向量混合召回、重排序；
3. 引用正确率与检索质量评测；
4. 扫描 PDF OCR；
5. PostgreSQL、HTTPS、备份恢复、监控告警与隐私策略；
6. 在获得合法授权后连接第三方内容平台 API。

## 6. 建设原则

- 不把 MVP 做成一次性页面；
- 不把业务逻辑绑定在单一模型厂商；
- 不用复杂 Multi-Agent 掩盖不稳定的基础流程；
- 不虚构爆款率、销量增长、实时热度或第三方平台授权；
- 不把真实密钥、客户资料、本地数据库或原始私密文档提交到仓库；
- 新功能必须有结构化数据、错误状态和最小测试；
- 比赛 Demo 与长期商业架构沿用同一条主线；
- 所有设计优先回答两个问题：农户能不能用，使用后的内容是否更好。
